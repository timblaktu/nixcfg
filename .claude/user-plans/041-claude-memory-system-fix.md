# Plan 041: Fix Claude Code Multi-Account Memory System

## Status: PENDING (awaiting approval after Phase 1 review)

## Background

Analysis document: `.claude/user-plans/040-claude-memory-system-analysis.md`

The refactoring in commit 16af8f0 (2026-06-07) introduced a regression: account
CLAUDE.md files were slimmed to 19-line pointer files referencing CLAUDE-SHARED.md,
but Claude Code does not follow markdown links. Shared rules are no longer loaded in
any account session unless the project has its own detailed CLAUDE.md.

## Why the Auto-Memory System Makes This Simple

Claude Code introduced a **built-in file-based memory system** (circa March-April 2026)
that stores persistent memories in `projects/<project-slug>/memory/` within the account's
config directory. This is a first-party Claude Code feature - not something configured in
this repo. It provides:

- Per-project memory files with YAML frontmatter (type, description)
- A `MEMORY.md` index file per project
- Automatic loading of relevant memories into session context
- Claude writes/updates these files as part of normal operation

**Timeline vs. the multi-account system:**
- 2026-02-16: Multi-account system created (`dd8f11e` - dendritic pattern migration)
- 2026-03-01: Double-loading fix (`fd4bdd5` - removed ~/.claude symlink, added per-account wrappers)
- ~2026-03-01: Multi-account memory architecture note added to CLAUDE-SHARED.md
- 2026-04-09: Earliest auto-memory file observed (MAX account, n3x project)
- 2026-06-07: The broken CLAUDE-SHARED.md refactor (`16af8f0`)

**The original multi-account system predates the auto-memory feature.** When it was designed,
CLAUDE.md was the ONLY place for persistent cross-session memory - hence the need for
writable CLAUDE.md files, `/nixremember` commands, and preserve-if-exists activation logic.

**Now that auto-memory exists, account CLAUDE.md no longer needs to be writable.** It can be
purely declarative configuration (rules, preferences, workflow instructions) managed by Nix
and regenerated on every `home-manager switch` - exactly like `settings.json` and `.mcp.json`.
Runtime memories go in the auto-memory system where they belong.

## Approach: Full Nix-Managed CLAUDE.md (Option C - Simplified)

Make account CLAUDE.md a pure Nix output:
- Generated per-account at eval time (shared content + account name interpolation)
- Overwritten on every `home-manager switch` (same as settings.json)
- No preserve-if-exists logic, no writability requirement
- Remove all "writable memory" infrastructure (`/nixremember`, `/nixmemory`, `memoryCommands`)

## Risk Assessment

**Low risk.** The auto-memory system (`projects/*/memory/`) is completely independent of
CLAUDE.md - it's managed by Claude Code's runtime, not by the activation script. Nothing
in this plan touches those directories. The only "loss" is the ability to `/nixremember`
append to CLAUDE.md, which the user confirms they never use.

## Design Decision: What Belongs in Account CLAUDE.md vs Project CLAUDE.md

After this fix:
- **Account CLAUDE.md** (Nix-managed): User preferences, shell/WSL rules, commit style,
  search tool preferences, general workflow instructions. Things true regardless of project.
- **Project CLAUDE.md** (per-repo, checked in): Architecture, build commands, technical
  learnings, project-specific conventions.
- **Auto-memory** (`projects/*/memory/`): Runtime discoveries, investigation state,
  project-specific facts learned during sessions. Managed by Claude Code itself.
